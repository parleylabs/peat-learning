A self-paced track for understanding the PEAT open-source mesh protocol — from high-level architecture down to how each repo is built and connected.

PEAT in one sentence. A decentralized mesh protocol that connects heterogeneous systems — phones, servers, sensors, embedded devices, AI models — into one coordinated whole, over any transport, even when the network is degraded or denied. The team's mental model is “TCP/IP for autonomy.” Built by Defense Unicorns, Apache-2.0.

1·5 · The Big Idea — Why PEAT Exists

The argument for PEAT, distilled from the whitepaper (peat/docs/whitepaper/, chapters 01–09). Read this before the architecture — it's the "why" everything else serves.

1 · The scaling crisis — math, not technology

Distributed multi-agent systems consistently plateau at ~20 nodes. The cause is O(n²) message complexity: in a full mesh, every node syncs with every other node.

NodesMessages/cycleImpact
10100manageable
20400heavy load
10010,000network collapse
1,0001,000,000physically impossible
The key line: “Better algorithms optimize within the constraint; they don't escape it. More bandwidth shifts the wall; it doesn't remove it.” The barrier is architectural, so the fix must be architectural. → implemented in hierarchy (Module 2) + peat-mesh routing/aggregation (Module 3).

2 · The standards paradox — the missing layer

MQTT, DDS, gRPC, ROS2, Modbus, CoT… all work, but all assume coordination “happens somewhere else — your application handles that.” Control is not coordination.

Application: domain logic
─────────────────────────────────────────────
??? COORDINATION: hierarchical orchestration ???   ← the layer PEAT fills
─────────────────────────────────────────────
Messaging: MQTT · DDS · gRPC · ROS2
Device control: Modbus · CAN · proprietary APIs
Network: TCP/IP · UDP · BLE · LoRa

The gap isn't in the standards we have — it's a layer that doesn't exist. PEAT bridges the existing standards (e.g. CoT/TAK, Module 5) rather than replacing them.

3 · The hierarchy insight — the reframe

Hierarchy isn't bureaucratic overhead — it's evolved communication optimization. Ant colonies coordinate millions without central servers. A leader tracking 3 summaries instead of 24 states is compression. Mesh is O(n²); hierarchy is O(n log n). For 1,000 nodes: mesh ≈ 500,000 connections, hierarchy (depth 4) ≈ 4,000.

UPWARD · aggregation
raw state summarized at each level — each tier gets only the granularity it needs
DOWNWARD · dissemination
intent translated into action — higher levels say what, lower decide how
LATERAL · coordination
peers sync within a level (handoffs, deconfliction) without involving the hierarchy

→ the three flows are Module 6 Trace C; the routing rule that enforces them is the Module 2 §2.4 snippet.

4 · Capability composition & emergence

Cells combine member capabilities into emergent ones no single node has — and coordinators task by requirement, not by node (the cell reallocates automatically on failure).

IndividualEmergent cell capability
Sensor + ComputeEdge AI processing
Multiple sensorsWide-area observation
Sensor + ActuatorSense-and-act loop
Heterogeneous sensorsMulti-spectral fusion

5 · Human-machine authority — “trust as data”

Humans sit within the hierarchy, not above it. Authority is replicated CRDT state (delegation/revocation propagate like any document), with configurable autonomy boundaries and graceful degradation (a disconnected node runs within last-known authority).

Level 0 Root      strategic decisions, all delegations
Level 1 Cluster   inter-formation coordination
Level 2 Formation mission assignment, resource allocation
Level 3 Group     tactical coordination, local autonomy
Level 4 Team      task execution, peer coordination
Level 5 Node      autonomous operation within constraints
Vocabulary drift: these whitepaper level names (Team/Group/Formation/ Cluster) are older; current code uses Cell / Cohort / Federation / Coalition (ADR-066). Same idea, different words.

6 · CRDTs instead of consensus

Consensus (Paxos/Raft/2PC) blocks until quorum — useless when partitions are the norm. PEAT assumes partitions: nodes operate autonomously and deterministically merge on reconnect.

Terminology: the whitepaper says DIL (Disconnected, Intermittent, Limited); the code/README says DDIL — the same, plus Denied (adversarial).

7 · The numbers (lab validation)

  • ~95–99% bandwidth reduction via aggregation (validation table: 93–99%) — from aggregation, not compression.
  • Phases: 2-node <1s/100% · 12-node 26s · 24-node 54s (6.1s mean) · simulated 1,000+ confirms O(n log n).
  • Maturity: TRL 4–5 (lab-validated, integration-demonstrated) — read as lab, not field-proven.

8–9 · Open imperative & why now

The TCP/IP lesson: infrastructure wins by enabling, not capturing. PEAT is Apache 2.0, IETF-spec'd, reference-implemented — coordination substrate, not product. Why now: autonomous deployment + edge-AI maturation + the reality that ubiquitous connectivity never arrived + multi-org coordination needs — and a standardization race where every proprietary adoption raises switching costs. Three integration depths: shallow (adapters/bridge, lowest risk) · medium (native capability advertisement, gradual migration) · deep (native peat-ffi/peat-lite, full authority model).

Scope: the whitepaper is a positioning + vision document. Treat its market claims as the project's thesis and its numbers as lab validation; the code modules are where you verify what's built today vs. roadmap.

1 · Architecture Overview

The shape of PEAT before touching code: the problem, the three phases, the five layers, the five repos, and the real dependency graph.

The problem

Edge and tactical environments are heterogeneous and the network is frequently degraded, disrupted, intermittent, or limited (DDIL). Phones, ESP32 sensors, edge AI servers, and robots all speak different protocols over different radios and often can't reach each other directly. PEAT gives them a common coordination layer that is offline-first (CRDT-based, no central server), transport-agnostic, and scales by hierarchy.

The three phases (runtime mental model)

1 · DISCOVERY
Nodes find each other — mDNS, BLE ads, static config, Kubernetes, or geographic clustering.
2 · CELL FORMATION
Discovered nodes form a cell by capability, compose those capabilities, and elect a leader.
3 · COORDINATION
Cells self-organize into a hierarchy; state aggregates up, commands flow down — without flooding.

A cell is the fundamental unit of organization. Cells form parent/child tiers (cell → cohort → federation → coalition) and keep operating autonomously when cut off.

There's no single "official layer count." The repo documents the architecture two complementary ways — knowing they're two lenses, not a contradiction, saves confusion.

The five repositories

peat/

Umbrella workspace. The SDK facade + schema + adapters + examples + spec. Crates: peat-protocol, peat-schema, peat-transport, peat-persistence, peat-ffi.

peat-mesh/

Network layer. QUIC/Iroh P2P, Automerge CRDT sync, discovery, topology. Ships the peat-mesh-node binary.

peat-btle/

BLE mesh. Bluetooth transport for phones, watches, sensors, MCUs (Linux/Android/iOS/macOS/ESP32).

peat-lite/

Embedded. no_std CRDT primitives + tiny wire protocol for 256 KB microcontrollers.

peat-gateway/

Control plane. Multi-tenant enrollment, CDC, identity federation, envelope encryption. Rust + SvelteKit UI.

The dependency graph (the literal truth)

This is what the Cargo.toml files declare. Arrows mean “depends on.”

Bold = the two crates an integrator cares about most. peat-protocol re-exports peat-mesh + peat-schema; peat-gateway uses peat-mesh as a library; peat-btle/peat-lite are optional features of peat-mesh.

Why the arrows changed direction. peat-mesh started life inside peat-protocol. ADRs (notably ADR-049 “HIVE mesh extraction” and peat-mesh ADR-0002) extracted it into its own repo so it could be versioned and published independently. After extraction, peat-protocol became a facade that depends on and re-exports peat-mesh. The old peat/docs/ARCHITECTURE.md (2025-01-07) still draws the pre-extraction arrow. Trust the manifests.

The modularity model (read this twice)

peat-lite and peat-btle are standalone leaf crates. The rest of PEAT depends on them; they depend on nothing in PEAT. Dependencies flow down into the edge crates, never out of them.
  • peat-lite is a pure leaf — zero PEAT dependencies (only heapless). A self-contained no_std CRDT library.
  • peat-btle is almost a leaf — its only PEAT dependency is an optional peat-lite. It does not depend on peat-mesh, peat-protocol, or peat-schema.
  • The core reaches outward to consume them, optionally — peat-mesh pulls in peat-btle (bluetooth) and peat-lite (lite-bridge); peat-protocol/peat-ffi pull in peat-btle per platform. Never the other way.
peat-protocol / peat-ffi / peat-mesh   ← the "core" stack
            │  (optional features point DOWN)
            ▼
       peat-btle    ← standalone BLE transport; only optional dep is peat-lite
            │  (optional)
            ▼
       peat-lite    ← pure leaf: zero PEAT deps, no_std, reusable anywhere

Why it matters (the framing captured during the IWRP proposal work): (1) independent reuse — each edge crate is adoptable on its own by projects that don't want the whole stack; (2) independent versioning/publishing (each is its own crates.io + Maven artifact); (3) it reaches targets the core can't — bare-metal MCUs, mobile — precisely because it doesn't drag in the heavier core; (4) it's what keeps the graph acyclic (the §1.5 cycle appeared the moment an edge crate pointed back up). The one-keyword tell: every PEAT arrow into peat-btle/peat-lite is marked optional = true.

Phase 2 — leader election is deterministic

Every node computes the same score for every candidate, so they all converge on the same leader with no vote-counting round. cell/leader_election.rs (~line 78):

let total = (compute * 0.30)        // inference / processing
          + (communication * 0.25)  // bandwidth, link diversity
          + (sensors * 0.20)        // sensor diversity
          + (power * 0.15)          // battery
          + (reliability * 0.10);   // historical uptime

A cell only becomes Ready when CellCoordinator::check_formation_complete() passes: min size, leader elected, roles assigned, capability coverage, readiness ≥ 0.7, and human approval for mission-critical formations. Emergent capabilities (composition/) mean a cell can do more than its members: camera + comms + range → ISR.

Phase 3 — the routing rule that defines the topology

hierarchy/router.rs (~line 86):

// Same cell → always allowed.
if from_cell == to_cell { return true; }
// Cross-cell direct messaging → always rejected.
//   (route up through your cell leader instead)
return false;

That single rule prevents an everyone-talks-to-everyone flood. State flows up (node→cell→cohort→federation→coalition), commands flow down, peers handle handoffs sideways. Aggregation is written against a trait (SummaryStorage), not a concrete DB — “traits over implementations” is everywhere in PEAT.

State, QoS, security

State is CRDT documents grouped into named collections; the AutomergeBackend (sync/automerge.rs) indexes docs by collection:id, fires observer channels on change, and orders deletions with a Lamport-counted tombstone (ADR-034). QoS classifies every message P1–P5 so contact reports preempt routine telemetry. Generic crypto (DeviceKeypair, FormationKey) lives in peat-mesh::security and is re-exported; peat-protocol adds the policy layer (RBAC, membership certs). Authority is “trust as data” — replicated CRDT state, so delegation/revocation propagate through the hierarchy like any document (see Module 1·5 §5 for the six-level authority model).

Step 2 · Formation auth — the handshake (the security crux)

Membership is proven by a shared formation key, never by mere reachability. FormationKey is derived from formation_id + a 32-byte secret via HKDF-SHA-256. The handshake runs over ALPN b"peat/formation-auth/1" before any state is exchanged:

1. initiator sends formation_id
2. responder replies with a fresh random NONCE (challenge)
3. initiator returns HMAC-SHA-256(nonce ‖ formation_id) using the formation key
4. responder verifies in CONSTANT TIME → admit / reject
sequenceDiagram
    participant I as Initiator (dialing node)
    participant R as Responder (formation member)
    Note over I,R: QUIC stream on ALPN "peat/formation-auth/1"
    I->>R: formation_id
    R->>I: fresh random nonce (the challenge)
    I->>R: HMAC-SHA-256(nonce ‖ formation_id), keyed by FormationKey
    Note over R: verify in constant time
    alt MAC valid
        R->>I: admitted — added to formation peer set
    else MAC invalid
        R--xI: rejected (wrong formation id or secret)
    end

Fresh nonce + formation_id in the MAC ⇒ non-replayable; a node in a different formation is rejected. AutomergeBackend does this automatically on every connection.

Step 3 · Leader election — deterministic, no consensus

score = compute·0.30 + communication·0.25 + sensors·0.20 + power·0.15 + reliability·0.10
ties → lexicographic node id

Every node computes the same winner from the same advertised capabilities — no vote round, split-brain safe. LeaderElectionManager runs Candidate → Leader | Follower; defaults: election 5s, heartbeat 2s, 3 missed (~6s) ⇒ re-elect. The spec adds a hybrid human-authority score + LeadershipPolicy (Rank/Technical/Hybrid/Contextual).

stateDiagram-v2
    [*] --> Candidate: start_election() — announce (round N)
    Candidate --> Leader: my score highest (tie → node id)
    Candidate --> Follower: better score observed
    Leader --> Leader: heartbeat every 2s
    Follower --> Candidate: 3 missed heartbeats — bump round
    Leader --> [*]: leave / partition

Step 4 · Roles — Leader elected, the rest assigned

CellRole: Leader (elected) + assigned Sensor, Compute, Relay, Strike, Support, Follower. RoleScorer::best_role_for_platform picks by fitness; required capabilities are blocking. (Deputy = 2nd-highest score, for fast failover.)

Step 5 · Confirm — the 6-gate readiness check

CellCoordinator::check_formation_complete() requires, in order: min size (3) · confirmed leader · every member has a role · capability coverage (Comms + Sensor) · readiness ≥ 0.7 · human approval if any capability needs oversight. Result → FormationStatus (Forming / AwaitingApproval / Ready / Failed).

3 · peat-mesh — the network layer

How bytes actually move: pluggable transports, Automerge CRDT sync over Iroh/QUIC, discovery, and topology. peat-mesh/

If peat-protocol is the “what” (cells, hierarchy, policy), peat-mesh is the “how” (connections, sync messages, persistence). It's re-exported, so apps rarely call it directly — but everything bottoms out here.

Two entry points

As a library — the builder

let mesh = PeatMeshBuilder::new(MeshConfig::default())
    .with_transport(iroh_transport)       // Arc<dyn MeshTransport>
    .with_hierarchy(hierarchy_strategy)    // Arc<dyn HierarchyStrategy>
    .with_discovery(discovery_strategy)    // Box<dyn DiscoveryStrategy>
    .build();
mesh.start().await?;

As a binary — peat-mesh-node

peat-mesh/src/bin/peat-mesh-node.rs (feature node) is the all-in-one node and the best assembly manual in the repo: derive keys from PEAT_FORMATION_SECRET via HKDF (ADR-062) → pick discovery (mDNS/k8s) → build a formation-gated Iroh endpoint → open the redb-backed AutomergeStore → run the AutomergeSyncCoordinator → start the mesh → dial peers → launch the Axum broker → graceful shutdown.

The CRDT sync wire protocol (the crown jewel)

One-byte-tagged messages in storage/automerge_sync.rs (~line 71):

#[repr(u8)]
pub enum SyncMessageType {
    DeltaSync          = 0x00,  // standard Automerge sync
    StateSnapshot      = 0x01,  // full doc.save() (LatestOnly)
    Tombstone          = 0x04,  // ADR-034 deletion
    SyncBatch          = 0x07,  // multiple docs in one message
    NegentropyInit     = 0x08,  // set reconciliation (ADR-040)
    NegentropyResponse = 0x09,
    NegentropyRequest  = 0x0A,
}
Why negentropy? Naive sync is O(n) in documents. Negentropy is a set-reconciliation protocol that finds the difference between two doc sets in O(log n) rounds by exchanging fingerprints — so peers learn which docs differ before sending any heavy data. Only then do they exchange Automerge deltas, persisting to redb as they go.
sequenceDiagram
    participant A as Node A
    participant B as Node B
    Note over A,B: Iroh QUIC stream (formation-gated)
    A->>B: NegentropyInit 0x08 — range fingerprints

    A->>B: NegentropyRequest 0x0A — doc IDs A is missing
    B->>A: StateSnapshot 0x01 — docs A lacks

The pluggable transport

Everything that moves bytes implements one trait (transport/mod.rs). Connection health is just RTT/loss thresholds:

pub enum ConnectionState {
    Healthy,   // RTT < 100ms, 0% loss
    Degraded,  // RTT >= 100ms OR loss > 5%
    Suspect,   // missed 2+ heartbeats
    Dead,      // connection confirmed closed
}

Three implementations plug in: Iroh QUIC (transport/iroh_mesh.rs, default, no relay servers unless relay-n0-hosted is enabled), peat-lite UDP (transport/lite.rs), and BLE (transport/btle.rs). Cross-transport bridging (ADR-059) goes through a Translator trait so a doc arriving over BLE can be forwarded over QUIC.

4 · peat-btle & peat-lite — the edge

Reaching the smallest, most disconnected devices: BLE mesh for phones/watches/MCUs, and no_std CRDT primitives for bare-metal sensors.

peat-btle — BLE mesh transport

When there's no WiFi/cell/satellite, the Bluetooth radio becomes the network. The design obsession is battery: where typical BLE-mesh SDKs burn 20%+ duty cycle (3–4 h watch life), peat-btle targets <5% and 18–24 h via batched sync windows and a “leaf nodes don't scan” hierarchy.

The CRDT merge rules are so small they're obvious — this is CRDTs without the jargon:

// lww.rs — last-writer-wins with a deterministic tiebreaker
fn should_accept(&self, ts: u64, node: NodeId) -> bool {
    ts > self.timestamp
        || (ts == self.timestamp && node.as_u32() > self.node_id.as_u32())
}
// counter.rs — grow-only counter merge = max per node (idempotent, commutative)
pub fn merge(&mut self, other: &Self) {
    for (&node, &c) in other.counts.iter() {
        let e = self.counts.entry(node).or_insert(0);
        *e = (*e).max(c);
    }
}

The 16-byte wire header (ADR-035): magic "PEAT", version, message type, flags, node id, seq num — then a payload. A Document (0x07) message type carries the universal peat_mesh::Document envelope, so a tiny sensor shares the same document model as a server. firmware/ targets ESP32 (xtensa toolchain); fuzz/ hardens the wire decoders.

How they connect (and the cycle that was broken)

The modularity model: these two are standalone leaf crates — the rest of PEAT depends on them, they depend on nothing in PEAT (beyond peat-btle's one optional link to peat-lite). That's deliberate (the IWRP-proposal framing): it lets each be reused on its own, versioned independently, and compiled for MCUs/mobile the core can't reach — and it's what keeps the graph acyclic. Every PEAT arrow into them is optional = true.
peat-mesh ──(bluetooth)──▶ peat-btle ──(peat-lite-frame)──▶ peat-lite
     └────────(lite-bridge)──────────────────────────────────▶ (leaf: no peat deps)
No cycles. peat-btle used to depend on peat-mesh (a mesh-translator feature), creating a cargo-breaking dependency cycle. ADR-059 Amendment 4 deleted that back-edge: peat-btle now exposes codec primitives (translator-codec) and peat-mesh implements the Translator on top. This is the saga behind the long comment block in peat/Cargo.toml.

5 · peat-gateway — the control plane

The enterprise side: onboard teams, federate identity, manage keys, and stream mesh events out — without ever sitting in the data path. peat-gateway/

Observes and manages; never in the data path. CRDT sync, blob transfer, and peer routing all stay in peat-mesh. The gateway is optional — meshes work without it — and adds enterprise manageability on top.

Entry point

src/main.rs is a clap CLI (Serve, MigrateKeys, LoadTest). The serve path is the spine:

The link to the mesh: MeshGenesis

The gateway manufactures the trust material autonomous nodes use. It creates and stores a MeshGenesis (peat-mesh's root authority) per formation, and maps enterprise roles to mesh trust tiers (src/tenant/):

use peat_mesh::security::{MembershipPolicy, MeshGenesis};
let genesis = MeshGenesis::create(&app_id, mesh_policy);
// roles → mesh tiers
MeshTier::Authority      => peat_mesh::security::MeshTier::Enterprise,
MeshTier::Infrastructure => peat_mesh::security::MeshTier::Regional,
MeshTier::Endpoint       => peat_mesh::security::MeshTier::Tactical,

Envelope encryption — the strategy pattern

Authority keys are encrypted at rest: a per-record DEK is wrapped by a KEK behind a pluggable trait (src/crypto/mod.rs):

#[async_trait]
pub trait KeyProvider: Send + Sync {
    async fn wrap_dek(&self, dek: &[u8]) -> Result<Vec<u8>>;
    async fn unwrap_dek(&self, wrapped: &[u8]) -> Result<Vec<u8>>;
}

Four backends: PlaintextProvider (dev), LocalKeyProvider (AES-256-GCM), AwsKmsProvider (AWS KMS), VaultTransitProvider (HashiCorp Vault). Callers never know where the key lives.

CDC — watching changes without being in sync

Change Data Capture streams CRDT mutations to enterprise systems by watching the mesh's document store (src/cdc/watcher.rs uses peat_mesh::sync::traits::DocumentStore + ChangeEvent). The CdcEngine fans CdcEvents out per-org to sinks: NATS JetStream, webhooks, Kafka, Redis Streams (at-least-once delivery).

Offline-first: a partition between steps 4 and 5 loses nothing — the CRDT update is committed locally and syncs when the link returns. QoS: a P1 contact report preempts P3/P4 traffic at every congested hop.

Trace B — discovery to a formed, led cell

Phase 1  peat-mesh/discovery (mDNS/k8s/static) → PeerConnector dials (Iroh QUIC)
         peat-protocol/discovery → candidate pool (60s)
Phase 2  a. CapabilityAnnounce on the reliable CellMessageBus (seq + ACK + retransmit)
         b. every node computes the SAME LeadershipScore → all converge on one leader
         c. CompositionEngine detects EMERGENT capabilities (camera+comms+range → ISR)
         d. check_formation_complete() → Ready → Phase 3

Why no consensus round? Scoring depends only on advertised capabilities, which every node receives. Same inputs → same winner, computed independently. Perfect for a flaky network.

Trace C — aggregation up, command down

UP    node track → CellDelta → HierarchicalAggregator → SummaryStorage → cohort/federation
DOWN  CommandCoordinator.issue_command() → CommandRouter resolves targets
      → cell leader forwards to members → ACKs tracked by TimeoutManager
      → conflicting commands resolved by ConflictResolver (LWW / highest-attribute)
The invariant that scales: the HierarchicalRouter only allows same-cell messaging or leader-mediated up/down routing — never cross-cell direct messages. More nodes ≠ more direct links. The protocol that works for 5 works for 1,000+.

Three log prefixes are how you read the mesh:

Four scenarios: (1) two nodes static peers · (2) three nodes — transitive gossip (charlie sees bravo via alpha) · (3) three nodes mDNS (zero-config, drop --peer) · (4) two Raspberry Pis + laptop (cross-compile aarch64). NodeId is deterministic from --name.

9 · The Protocol Specifications

The normative layer — five IETF RFC-style specs (docs/spec/, RFC 2119 MUST/SHOULD/MAY) that define PEAT precisely enough to reimplement. Read order: 003 → 001 → 002 → 004 → 005.

Dated 2025-01-07 — some details evolved. Where a spec disagrees with a newer ADR or the manifests, the newer source wins; but the specs are the best single description of the protocol's shape.

001 · Transport

Wire formats & lifecycle. PeerId = SHA-256(Ed25519 pubkey); big-endian. QUIC/Iroh primary (stream-id ranges: control/CRDT/app/user). UDP bypass is unauthenticated by default — Signed mode mandatory on untrusted links. Peat-Lite (0xCAFE, CRC-16) & BLE (GATT 0x1826).

002 · Sync

Exclusively Automerge + Negentropy. Map=LWW, List=RGA, Counter=PN. Deterministic conflict resolution (Lamport ts + actor-id tie-break). Deletes are tombstones (7-day retention).

003 · Schema

Protobuf tactical entities (Beacon, Mission, Capability) in versioned packages; strict evolution rules. Bidirectional CoT/TAK mapping (a-f/a-h/a-n/a-u + G/A/S/U, <__peat> extension).

004 · Coordination

Normative Module 2·5: cell state machine, handshake, election weights (0.30/0.25/0.20/0.15/0.10) + hybrid human-authority, emergent-capability patterns, the three flows, partition merge.

005 · Security

Ed25519 identity (DeviceId=SHA-256(pubkey)), challenge-response with version negotiation (ADR-065), RBAC + access levels. FIPS 140-3 (ADR-060): AES-256-GCM, ECDH P-256/384, HKDF/HMAC-SHA-256 — supersedes ChaCha20. Hash-chained audit log; MLS for forward secrecy.

Eight facts to carry from the specs

  • PeerId/DeviceId = SHA-256(Ed25519 pubkey) — one derivation, used everywhere.
  • Automerge + Negentropy is the only sync stack; conflict resolution is deterministic.
  • QUIC/Iroh primary; stream-id ranges partition control/CRDT/app traffic.
  • UDP bypass unauthenticated by default — Signed mode mandatory on untrusted links.
  • Formation key = HMAC-SHA-256 proof-of-knowledge, separate from the Ed25519 device challenge.
  • Leader election is hybrid (technical + human authority), deterministic, no consensus round.
  • Emergent capabilities are formal patterns driving capability-based tasking.
  • Crypto is FIPS 140-3 (AES-256-GCM / P-256-384 / Ed25519) — newer than prose docs that say ChaCha20 / X25519.
