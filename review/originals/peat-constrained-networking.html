When the network won't cooperate

PEAT normally rides on QUIC, a modern internet protocol that quietly assumes three luxuries: every device has a network address, bandwidth runs into the millions of bits per second, and a reply comes back within a few thousandths of a second.

A great deal of the work PEAT is built for, though, happens where none of those luxuries are on offer — a ridgeline with no cell signal, a vessel far past the last tower, a disaster zone where the infrastructure is rubble. This track follows the mesh as its link thins out rung by rung, from hundred-megabit Wi-Fi all the way down to a single satellite burst of fewer than two thousand bytes that a device might only get to send once an hour.

01 · QUIC is not your address

The whole problem looks enormous until you separate two ideas that most networking quietly merges together: who a device is, and how you happen to reach it today.

In PEAT, a device's identity is its NodeId — a 256-bit fingerprint computed from the device's own public cryptographic key. Because it comes from a key only that device holds, the name is globally unique and effectively impossible to forge, and it stays the same no matter how the device connects. QUIC only looks like it owns "addressing" because the library PEAT uses for it, Iroh, builds its own identity from that very same kind of key, so on an ordinary internet connection several layers stack neatly onto one piece of software.

Pull those layers apart and a simple rule appears: the name never changes, while the delivery address is whatever a given radio happens to use today.

Move house, keep your name. Each transport carries one small lookup table from NodeId to its own kind of address — a satellite modem's IMEI number, a LoRa radio endpoint, a short BLE id.
name — the NodeId (never changes) address — per transport (changes)
What this buys you: moving PEAT onto a new, non-internet radio never means reinventing identity. It means writing one small address-translation for that radio — a contract the transport design (ADR-032) already spells out.

02 · The three layers

QUIC does three separate jobs at once, and on a normal network you never notice the seams. Off-grid, those jobs come apart, and each one needs a different replacement. This stack is the map for everything that follows.

The first two layers are mostly plumbing. They take real engineering, but the shape of the work is well understood, and PEAT already has the seams for it — a transport interface and a separate code crate for each radio. The third layer is where the genuine design problem lives, because making independent copies of data agree is exactly the thing that QUIC's behaviour was quietly helping with. 03 · Why CRDTs change everything

QUIC spends a lot of effort guaranteeing that every byte arrives exactly once, in the right order, with nothing lost. The data PEAT carries mostly doesn't need that guarantee, and understanding why is the key that opens the whole constrained world.

PEAT stores its shared state in CRDTs — Conflict-free Replicated Data Types. The mouthful matters less than one plain property: a CRDT update can be applied in any order, and applying the same update twice does nothing extra. Picture two people each keeping a copy of a shopping list and mailing postcards back and forth — "add milk," "cross off eggs." It makes no difference if a postcard shows up late, shows up twice, or arrives before one written earlier. Fold them all in, and both lists end up identical.

Late, duplicated, out of order — it all converges to the same state. The reliable, in-order delivery QUIC works so hard for turns out to be machinery this data never required.
✉️ = one CRDT update (a delta) a device holding its own copy

Compare that to reading a recipe aloud over a crackly phone line, where a single dropped or swapped word ruins the dish — that is the kind of protocol that genuinely breaks on a twenty-second satellite hop. Because PEAT behaves like the postcards instead, it can shed QUIC's delivery guarantees and lean on the mathematics of the data itself, which holds up even when the link loses, repeats, and delays messages.

There is a practical bonus on the thinnest links. You can send the current value of something rather than its whole history. If a sensor reading changed fifty times while a device was out of contact, you transmit one number and one timestamp, not fifty updates — and peat-lite's data types are designed as small, bounded values precisely so this collapse is cheap.

04 · Transports as a PACE ladder

Every radio strikes a different bargain between reach, speed, power, delay, and cost. PEAT registers all the ones a device carries and chooses among them with a planning method borrowed from the military: Primary, Alternate, Contingency, Emergency.

The pattern across radios is consistent — the farther a signal needs to travel, the less data it can carry. Bluetooth Low Energy moves a couple of megabits per second but only reaches a few hundred metres. LoRa stretches from seven to nearly ninety kilometres, paid for by dropping to mere kilobytes per second. Iridium's satellite service reaches anywhere on Earth, pole to pole, but a single message is capped near 1,960 bytes, takes five to twenty seconds to land, and costs roughly ten cents to send.

When the primary link dies, selection steps down a rung — the same mesh on a worse-but-working radio. A data type can set a floor (min_pace_level) so that urgent commands ride even the bottom rung while routine telemetry waits for something better.

05 · Dialects + the interpreter

The same software cannot run everywhere. The full version of PEAT's sync engine, Automerge, needs around ten megabytes of memory and a full operating system. A typical sensor microcontroller has about 256 kilobytes and no operating system at all — smaller by a factor of roughly forty.

The answer is two dialects of the same language, with a translator in the middle. Tiny devices speak peat-lite, a compact set of the same conflict-free data types squeezed into a 16-byte message header. A better-resourced device — a phone, a Raspberry Pi, a small Linux box — sits at the boundary and is fluent in both, translating the lightweight edge dialect into full Automerge for the richer mesh and back again.

No constrained radio ever has to make a ten-megabyte engine survive a slow link. The edge sends a small update; the gateway does the heavy merging on its behalf.

06 · Anti-entropy: one message an hour

Left alone, two copies of shared data drift apart as each picks up changes the other hasn't seen. "Anti-entropy" is the periodic ritual that pulls them back together: the two copies compare notes and trade whatever is missing. On a satellite link, where one exchange costs about a dime, takes up to twenty seconds, and may be the only one you get that hour, every byte of that comparison has to earn its place.

Iridium's hardware quietly hands you a gift here. A single satellite session does two things in one pass: it sends your outgoing message and delivers any message waiting for you. That is a complete, if slow, round trip for the price of one. So you design each session to carry a full reconciliation — your outgoing burst includes both your recent changes and a short summary of everything you currently hold, and the reply comes back carrying exactly the changes you were missing.

The summary is the clever part: a device describes what it has compactly enough that the gateway can work out what to send back, all within a single 1,960-byte budget.
up: your changes + summary down: what you were missing

If the device remembers

It keeps a small "reading log" — the latest update number it has seen from each peer, about 8 bytes apiece. A cell of up to a hundred devices fits in one satellite message with room left for real data.

If it forgets on reboot

Many tiny devices keep nothing across a restart, so the device simply sends its own latest number and the time it last made contact. The gateway replies with everything that changed since then, newest values first.

At larger scale

A summary for a thousand devices would be too big, so the device carries a full log only for its own small group and a single fingerprint for each tier above it, reusing the summaries the hierarchy already produces.

On faster radios

LoRa can afford a couple of exchanges, so it can use a compact "difference sketch" sized to the expected gap, letting both sides find what differs without listing everything they share.

The hard part is deletion. If you only send current values, "this was deleted" looks the same as "I never heard of it." The fix is a small marker that records a deletion, kept around at least as long as the slowest device might stay out of contact, with the gateway as the keeper of record. This is active work in PEAT today (issue #857).

07 · Router, not broker

A design where many sensors all sync through one main station sits dangerously close to the kind of central broker PEAT was built to avoid. The distinction that keeps it honest is the difference between a place where roads meet and a seat of government.

The gateway is a meeting point, not a ruler. It holds the most complete copy of the data simply because that is where the satellite traffic happens to land — the same way a mountain town ends up busy because it sits on the only pass. As long as the gateway only merges data (a deterministic step that any device with the same updates would arrive at identically) and passes it along, it holds no special authority. The danger line is crossed only if the gateway starts deciding disputes by its own judgment instead of by the shared merge rule.

The test is simple: remove the main station and watch. A broker was the network, so the network dies with it. In PEAT the data already lives, copied, across the devices — another capable device takes over the gateway role and the sensors reconnect to it.

08 · Two graphs, one system

It helps to see that any deployment really has two different maps drawn over the same devices. One map shows which devices can physically reach each other. The other shows which devices actually exchange state with each other. They are not the same map, and the gap between them is a deliberate choice driven by what each link costs.

Use the buttons to switch between the two maps for one small system — three sensors near each other, a gateway with a satellite dish, and a command post on the far side of the satellite.

What can reach what: the three sensors are in radio range of each other and of the gateway, and the gateway can reach the command post only by way of the satellite.
local radio link — cheap satellite hop — slow & paid full two-way sync summary only
The point you'd miss otherwise: you could have all five devices sync with each other as one flat mesh across the satellite — nothing stops you. You wouldn't, though, because every peer-to-peer exchange would then cost two satellite hops, around forty seconds and two paid messages. So the dense, everyone-with-everyone syncing stays on the cheap local links, and the satellite carries only a compact summary up and commands down. "Decentralized" describes who holds authority over the data, not a promise that every device chats with every other.

09 · One node, three dials

A mute telemetry sensor and a fully coordinating peer run the same software — they differ only in how three independent dials are set. Here is one real system; click a role to see where that device sits, how its dials are turned, and how data flows around it.

Each dial runs low → high. Reachability: how many links, how far · Participation: how much shared state it reads & writes · Authority: its command rights.

Sensor A — mute sensor. It sits inside the local cell on short-range radio (BLE or similar) with no uplink of its own. It publishes its own readings and never coordinates with its peers, at the lowest authority; any command for it arrives down through the gateway. The wider mesh effectively begins there.
local sync — cheap satellite path — paid amber ring = selected device
Notice that the everyday, high-volume coordination all happens on the cheap local links among A, B and their neighbours, while the satellite carries only the summary up and commands back down — exactly the choice section 08 was pointing at. Changing a device's job is a matter of turning these three dials, never swapping its software.

10 · Match coordination to cost

The single most useful design rule for these systems follows directly from the last two sections: put the boundaries of your hierarchy wherever a link gets expensive.

Coordination that is almost free over a local Bluetooth link becomes punishing over satellite. Each back-and-forth between two peers turns into two satellite hops — roughly forty seconds of delay and two paid messages — so a chatty exchange that would finish instantly nearby could take an hour and several dollars across the sky. The way out is to let devices that are physically close form a tight local group and coordinate freely there, and to let the expensive satellite link carry only the distilled result upward and commands downward.

Keep the dense traffic on the cheap side of the cliff; send only the aggregate across it. The satellite link should sit between groups, never inside one.
cheap local links summary across the cliff cost cliff (tier boundary)
The rule of thumb: place each tier boundary at a jump in link cost. That one decision often decides whether the satellite bill is survivable.

11 · Configuring both

Most real deployments need both kinds of coordination at once: a shared picture that everyone agrees on, and direct orders aimed at specific devices. The good news is that this choice lives on each collection of data, not on the device, so a single system runs both side by side.

You define each collection with the profile its data deserves. Positions and sensor readings want a "shared picture" profile — broadcast to the whole group, merged by everyone. Orders want a "directed tasking" profile — sent only to chosen recipients, writable only by an authorized commander, and acknowledged on receipt.

Shared-picture profile

positions · tracks · readings

tracks:
  delivery: broadcast   # all members merge
  crdt: or_set | lww
  priority: background
  ttl: 5m
  min_pace: emergency

Directed-tasking profile

orders · tasking

orders:
  delivery: targeted    # chosen recipients
  target: "role:scout"
  crdt: command_log
  write_role: operator
  priority: critical
  require_ack: true

The mute sensor from section 09 needs no third profile. It simply uses each of these in one direction only: it publishes into a shared-picture collection it never reads back, and it listens to a tasking collection it never writes to.

One real gap remains. peat-lite's current data types are safe for facts and counts, but none is safe for orders. Settling two conflicting commands by "newest wins" could quietly fire the wrong one. A tasking collection needs a command_log type that records orders in sequence and marks superseded ones plainly, so a person can always see the chain. The shared-picture half works today; the targeted half is the ADR-046 work already underway; the command type still has to be designed.

12 · Gaps & where to go next

A quick honest map of what is solid, what is being built, and what still needs design — plus the doors out of this track.

built — works today in flight — being built to design — open

Open questions worth your attention

  • The command-log data type — recording orders in sequence and marking superseded ones, safe for real-world action under human authority.
  • The summary's exact format — how the "reading log" and the "everything since this time" forms share one compact message.
  • How long to keep deletion markers — tied to the slowest device's time away, with the gateway as keeper of record (issue #857).
  • Encryption choice — today's constrained transports specify a cipher that doesn't meet PEAT's required (FIPS) list; an object-security approach would fix this and protect each message end to end through an untrusted gateway.
