# Claim Ledger — `learning/09-protocol-specs.md` (Module 9 · Protocol specs)

Audited against `learning/review/ground-truth/*` (peat HEAD `35d0f11`, peat-mesh `00ab0c9`/rc.42,
peat-btle `3d70f48`/0.4.0, peat-lite `7a8a8fb`/0.2.5, peat-gateway `8d16824`, peat-node `4e1b5c8`)
**and the actual spec files in `peat/docs/spec/` + ADRs in `peat/docs/adr/`** (read directly).
Verdicts: Verified · Outdated · Misleading · Oversimplified · Wrong · Unverifiable · Speculative-unlabeled.

**Headline finding for this module:** Module 9 describes the spec layer mostly *faithfully to the specs as
written* — but two structural problems run through it. (1) Several spec values that the module repeats are
**contradicted by shipped code** (Device ID 32 B vs code's 16-byte truncation; RBAC `Role` enum
Observer/Member/Operator/Leader/Supervisor vs code's `Leader/Member/Observer/Commander/Admin`). The module
presents the spec value as if it were the implemented contract. (2) The module's central "the specs are stale,
they still say ChaCha20, the code is newer" narrative is **itself out of date** — spec 005 was amended
**2026-05-18 (rev 0.2.0)** to AES-256-GCM and is now FIPS-clean. The module repeatedly tells the reader to go
find the ChaCha20 in spec 005; it is no longer there.
