# Claim Ledger — `learning/06-data-flows.md` (Module 6: Cross-Cutting Data Flows)

Audited against `learning/review/ground-truth/*.md` and the live `peat/` source tree
(`peat-protocol`, `peat-schema`). Verdicts: **Verified · Outdated · Misleading · Oversimplified ·
Wrong · Unverifiable · Speculative-unlabeled**.

This module is unusually well-grounded: most cited `path` references resolve and the leader-election
formula is quoted *verbatim* from code. The defects are concentrated in (1) one fabricated enum
variant name, (2) a non-existent method name, (3) cross-crate identity/transport oversimplifications
that hide the peat-lite vs peat-mesh reality, and (4) the recurring unlabeled "1,000+ node" scaling
claim.

## Cross-cutting notes

- **Best-grounded module audited so far.** Nearly every `path` reference resolves, and the
  leader-election scoring formula and weights are quoted exactly from `leader_election.rs:101-106`.
  Trace B's "deterministic, no consensus" — a named high-risk claim — is **correct**.
- **Two genuine code defects** a skeptical engineer would catch on first grep:
  `ConflictResolver` policy **`HighestAttributeWins`** (does not exist; real variants are
  `HIGHEST_PRIORITY_WINS` / `HIGHEST_AUTHORITY_WINS`), and **`CommandRouter.resolve_targets()`**
  (real method is `resolve_target` singular / `get_routing_targets`).
- **Recurring unlabeled scaling claim.** "Works for 1,000+ … for exactly this reason" presents a
  single-machine-simulation ceiling (1023 limit, open validation epics #724-#727) as an
  architectural certainty. Same pattern flagged in the peat.md ground truth (§9, gap list).
- **Two distinct leader-election implementations** exist (peat-protocol `leader_election.rs`
  capability-weighted vs peat-mesh `dynamic_strategy.rs` mobility/resource/battery-weighted). The
  module cites the correct one for cell formation but should note the mesh-level one to avoid
  implying a single global algorithm.
- **No FIPS or vendor-name violations introduced** by this module. CoT / TAK / WebTAK / ATAK are
  protocol/product *protocol* names (CoT) and acceptable; no consumer vendor names misused. No
  crypto primitive is misstated (Module 6 doesn't assert crypto algorithms).
- **Diagrams lack the required legend / shipped-vs-proposed key** (prompt §8, DoD). Factually the
  depicted flows are all shipped, but the labeling rule is unmet.
- **"256KB device" figure** is an unlabeled design target, not an enforced/measured limit
  (gt peat-lite §9).
