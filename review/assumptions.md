# Peat Curriculum — Assumptions Register

Every premise the curriculum (or this audit) relies on, where it is relied on, whether it holds at
the audited HEADs (`learning/review/ground-truth.md` §1), and the consequence if it is false.
Seeded with the premises the prompt §5 names, expanded with the assumptions surfaced across the 14
ledgers and 6 ground-truth audits.

| # | Assumption | Where relied on | Holds now? | Consequence if false |
|---|---|---|---|---|
| A1 | **A gateway with backhaul is always reachable** | constrained track satellite/PACE narrative; sensor→command-post trace (00, 06); any "gateway merges and forwards" framing | **No (and a category error).** peat-gateway is a control plane that **never sits in the data path**; the data-plane "gateway" the track means is a peat-mesh relay *node*, and its reachability is exactly what DDIL operation must survive (GT §7). | The whole off-grid spine collapses: if the curriculum assumes the control-plane gateway relays/translates/merges, it teaches a capability that does not exist. Rename to "hub/relay node" and treat reachability as intermittent. |
| A2 | **Edge nodes are ephemeral and keep no state across reboot** | constrained cn6 snapshot-since-T card; reconnect/re-sync claims (06, 08) | **Partially.** peat-node persists to disk and re-syncs via CRDT convergence; peat-lite is `no_std` with optional persistence; peat-btle reconnect re-delivery of pending CRDT state is **open** (#73, In-flight). The snapshot-since-T reconciliation scheme that depends on this is **Speculative** (not implemented). | If presented as shipped, a reader expects clean post-reboot reconciliation over SBD that does not exist. Tombstone GC + retention (#136) also affect what re-syncs after long offline windows. |
| A3 | **Iridium `+SBDIX` does MO+MT in one pass** (one session sends and receives) | constrained cn6 "Iridium hands you a gift" | **Unverifiable as a Peat fact** — it is an external Iridium hardware behavior, and **Peat has no SBD code** (peat-sbd is Proposed, ADR-051). | If stated as Peat behavior, the latency/cost arithmetic ("~40s / two paid messages") is doubly unfounded: it rests on an uncited hardware claim AND an unbuilt transport. Cite the Iridium source; mark as a hardware assumption. |
| A4 | **A node persists a version vector / per-peer "reading log"** | constrained cn6 anti-entropy cards ("8 bytes apiece", "100 devices in one message") | **No.** No version-vector reconciliation is implemented anywhere; shipped anti-entropy is **negentropy** over QUIC (GT §3). | The byte-budget arithmetic and the entire per-satellite-pass digest scheme are teaching-only design; presenting them as Peat's mechanism is false. |
| A5 | **Tier boundaries can be placed at the link-cost cliff in practice** | constrained cn8/cn10 cost-cliff topology; "dense local sync cheap, satellite carries summaries" | **Speculative.** Depends on the unbuilt SBD/LoRa transports and the speculative digest scheme; hierarchy sizing (Cell 4-13, Cohort 2-4 cells) is design-intent doc-comment, **not enforced** (GT §5). | The clean tier/transport alignment is aspirational; real deployments have no enforced sizing and no satellite transport to anchor the cliff. |
| A6 | **The leaf hierarchy tier is named `Platform` (ADR-066 is the current vocabulary)** | hub m25, mbig §5; 00b §5; 01 §1.2; 02 #29; 02b §2·5.1; 09 §9.4 | **No.** Shipped enums use `Node` as the leaf (peat-mesh, peat-protocol); peat-btle is fully legacy Squad/Platoon/Company; ADR-066 is **Proposed** and the rename is mid-flight (#904/#968) (GT §5). | The most-catchable error in the curriculum: a reader greps `HierarchyLevel` and finds `Node`, not `Platform`. Undermines trust in every other vocabulary claim. |
| A7 | **There is one identity derivation used across the stack** ("NodeId = SHA-256 of Ed25519 key, everywhere") | hub m9 "Eight facts"; 09 §9.6 fact 1; constrained cn1; 00 sensor→command-post trace | **No.** Four schemes: DeviceId SHA-256[..16]; mesh transport NodeId = String; btle NodeId = BLAKE3[..4] u32; lite NodeId = bare u32; peat-node = raw iroh EndpointId (GT §4). | A reader greps and finds four different identity schemes. The cross-crate identity bridge is non-trivial and partly unverified (`btle_to_peat_node_id` not found). |
| A8 | **Crypto is FIPS-validated** (140-3) | hub m9 fact 8; 09 §9.5; 08 cipher caveat | **Algorithms only.** AES-256-GCM/ECDH-P256/Ed25519/HKDF/HMAC are FIPS-**approved** but run in pure-Rust RustCrypto crates, **not a CMVP-validated module** (peat-btle#75 open). For a real FIPS 140 boundary the KMS/Vault HSM path is required (GT §6). | An auditor at a defense prime will reject "FIPS 140-3" for software not in a validated module. Must distinguish approved-algorithm from validated-module. P-384 is also not in code. |
| A9 | **The READMEs/specs reflect shipped behavior** (so the curriculum can quote them) | wherever modules echo README role names, ChaCha20 cipher, peat-lite/BLE wire formats, "25 RPCs", peat-mesh versions | **No.** READMEs are stale on crypto (ChaCha20/X25519), RBAC role names (Operator/Supervisor), peat-mesh version (0.3.2 vs rc.42), and peat-node RPC count (25 vs 28/27); spec values diverge from code on Device ID width, Role enum, peat-lite/BLE wire formats, leader-election weights (GT §5, §6, §9). | Quoting docs without checking code propagates exactly the errors this audit catches. Operating principle must be **code > current ADRs > specs/guide/README**. |
| A10 | **MLS provides forward secrecy in shipped cells** | hub m9 §005; 09 §9.5; README §Layer-4; 08 "no MLS" | **No.** MLS is **not implemented** (no `openmls`/`mls-rs`); Proposed (ADR-044, carries pre-FIPS ChaCha20) (GT §6). | Presents an unbuilt forward-secrecy mechanism as shipped — a serious security overclaim for the audience. Today's group-key rotation is leader-distribution. |
| A11 | **SBD and LoRa transports exist** (so PACE, satellite topology, anti-entropy bursts work) | constrained track spine (cn4-cn12); 00 "over any transport"; 01 "any transport works" | **No.** peat-sbd (ADR-051) and peat-lora (ADR-052) are **Proposed-only — no crate, no module, no impl**; `LoRa`/`Satellite` enum members resolve to "no transport registered" (GT §2). | The constrained track's entire premise is built on transports that do not exist. Every figure (1,960B SBD, 7-87km LoRa) is an external hardware spec, not a Peat capability. |
| A12 | **`command_log` is a near-term CRDT/collection-config option** | constrained cn11 directed-tasking YAML | **No.** `command_log` exists **nowhere** in any repo (GT §3). Commands are ordinary JSON docs in a `commands` collection. | Presents a CRDT that has not even been designed as a config knob. The honest framing (cn11 warn) says the command type "still has to be designed" — keep that, drop the YAML. |
| A13 | **QoS priority is enforced end-to-end** (P1 preempts, "<5s P1") | 06 Trace A; hub m6; 02 #22; 08 §8.4 | **Partially.** QoS classes/allocation/eviction ship and peat-node orders by class, but **cross-class wire-level preemption is NOT enforced in v1** and "<5s P1" is a target (enforcement In-flight) (GT §8). | A reader expects guaranteed P1 latency the system does not deliver yet. Frame as "prioritizes/orders," enforcement In-flight. |
| A14 | **Leader election uses the 0.30/0.25/0.20/0.15/0.10 weights at the mesh layer** | hub m2; 02b §2·5.5; 09 §9.4 | **No (wrong layer).** Those are the peat-protocol cell-formation / spec-004 weights; **peat-mesh's runtime score uses a different formula** (mobility/CPU/mem/battery, GT §5). Two distinct deterministic elections exist. | Implies a single global algorithm. A reader verifying against `dynamic_strategy.rs` finds different inputs. Label which layer each formula belongs to. |
| A15 | **The HTML curriculum renders in an air-gapped environment** | hub diagrams (Mermaid) | **No.** The hub loads Mermaid from `cdnjs.cloudflare.com` — diagrams will not render offline (the target audience's environment). | Diagrams silently break for exactly the air-gapped/contested-comms audience the material targets. Inline or pre-render to SVG. |
