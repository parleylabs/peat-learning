Start Here

A self-paced track for understanding the Peat open-source mesh protocol — from high-level architecture down to how each repo is built and connected. It is written to survive a skeptical staff engineer reading it with the Peat source open in another window.

Peat in one sentence. A decentralized mesh protocol that connects heterogeneous systems — phones, servers, sensors, embedded devices, AI models — into one coordinated whole, over the transports it supports, even when the network is degraded or denied. The team's mental model is “TCP/IP for autonomy.” Built by Defense Unicorns, Apache-2.0.

1·5 · The Big Idea — Why Peat Exists

The argument for Peat, distilled from the whitepaper (peat/docs/whitepaper/). Read this before the architecture — it's the "why" everything else serves. Treat the whitepaper as positioning and design intent (Speculative unless a module later confirms the code), not as proof of field results.

1 · The scaling crisis — math, not technology

Distributed multi-agent systems consistently plateau at ~20 nodes. The cause is O(n²) message complexity: in a full mesh, every node syncs with every other node.

NodesPairwise links (n·(n-1)/2)Impact
1045manageable
20190heavy load — the empirical ceiling
1004,950network collapse
1,000~500,000physically impossible as a full mesh
The key line: “Better algorithms optimize within the constraint; they don't escape it. More bandwidth shifts the wall; it doesn't remove it.” The barrier is architectural, so the fix is architectural. Shipped as the hierarchy module (Module 2) plus the routing rule that forbids cross-cell direct messaging — that rule is what bends the curve.

2 · The standards paradox — the missing layer

MQTT, DDS, gRPC, ROS2, Modbus, CoT… all work, but all assume coordination “happens somewhere else — your application handles that.” Control moves bytes; coordination decides who should act. Peat supplies the coordination layer and bridges the existing standards (notably CoT/TAK, Module 5) rather than replacing them.

Application: domain logic
─────────────────────────────────────────────
COORDINATION: hierarchical orchestration          ← the layer Peat fills
─────────────────────────────────────────────
Messaging: MQTT · DDS · gRPC · ROS2
Device control: Modbus · CAN · proprietary APIs
Network: TCP/IP · UDP · BLE · (LoRa, proposed)

3 · The hierarchy insight — the reframe

Hierarchy here is communication compression, not bureaucracy. A leader tracking 3 child summaries instead of 24 raw states is doing aggregation. The complexity claim — full mesh is O(n²), hierarchy is O(n log n) — is an analytical/design result from the whitepaper's connection table (full mesh ~9,120 links vs hierarchical ~384 at scale), not a measured runtime benchmark. A second whitepaper figure (500,000 vs ~4,000 at depth 4) is a looser model that disagrees with the first by roughly 10×, so treat the exact numbers as illustrative.

UPWARD · aggregation
raw state summarized at each level — each tier gets only the granularity it needs
DOWNWARD · dissemination
intent translated into action — higher levels say what, lower decide how
LATERAL · coordination
peers sync within a level (handoffs, deconfliction) without involving the hierarchy

→ the three flows are Module 6 Trace C; the routing rule that enforces them is the Module 2 routing snippet. Autonomy under human authority: higher tiers express intent, lower tiers retain how-to-execute discretion within delegated bounds.

4 · Capability composition & emergence

Cells combine member capabilities into emergent ones no single node has, and coordinators task by requirement, not by node (the cell reallocates on failure). The capability taxonomy in code is CapabilityType = SENSOR, COMPUTE, COMMUNICATION, MOBILITY, PAYLOAD, EMERGENT Shipped (peat-schema/proto/capability.proto). There is no Weapon type — weapons fall under PAYLOAD.

5 · Human-machine authority — “trust as data”

Humans sit within the hierarchy. The design intent is that delegation and revocation propagate like any replicated document, so a disconnected node keeps operating within its last-known authority — autonomy under human authority. The whitepaper draws a six-level ladder, but that ladder is whitepaper-only; the shipped axis is a five-value enum.

Whitepaper ladder (Speculative)Shipped AuthorityLevel enum Shipped
Level 0 Root … Level 5 Node (six levels)UNSPECIFIED, OBSERVER, ADVISOR, SUPERVISOR, COMMANDER (five) — peat-schema/proto/node.proto
Three different axes — do not conflate them. AuthorityLevel (human-machine teaming, above) is distinct from the RBAC Role enum (Leader, Member, Observer, Commander, Admin) and from CellRole (the capability axis: Leader, Sensor, Compute, Relay, Strike, Support, Follower). The README's "Operator/Supervisor" RBAC names are wrong — those strings are not in the enum.

6 · CRDTs instead of consensus

Consensus (Paxos/Raft/2PC) blocks until quorum — a poor fit when partitions are the norm. Peat assumes partitions: nodes operate autonomously and merge deterministically on reconnect via Automerge CRDTs. Shipped — this is the strongest part of Peat.

Terminology: the whitepaper says DIL (Disconnected, Intermittent, Limited); the code/README says DDIL — the same, plus Denied (adversarial).

8–9 · Open imperative & why now

The TCP/IP lesson: infrastructure wins by enabling, not capturing. Peat is Apache-2.0, IETF-spec'd, reference-implemented — coordination substrate, not product. Three integration depths (with author LOC/latency estimates, not measured): shallow (adapters/bridge, ~500–1000 LOC) · medium (native capability advertisement, ~2000–5000 LOC) · deep (native peat-ffi/peat-lite, ~5000+ LOC).

1 · Architecture Overview

The shape of Peat before touching code: the problem, the three phases, the layer lenses, the repos, and the real dependency graph.

The problem

Edge and tactical environments are heterogeneous and the network is frequently degraded, disrupted, intermittent, or limited (DDIL). Phones, ESP32 sensors, edge AI servers, and robots speak different protocols over different radios and often can't reach each other directly. Peat gives them a common coordination layer that is offline-first (CRDT-based, no central server), transport-agnostic across the transports it supports, and scales by hierarchy.

The three phases (runtime mental model)

1 · DISCOVERY
Nodes find each other — mDNS, BLE ads, static config, or Kubernetes EndpointSlice. Shipped
2 · CELL FORMATION
Discovered nodes form a cell by capability, compose those capabilities, and elect a leader. Shipped
3 · COORDINATION
Cells self-organize into a hierarchy; state aggregates up, commands flow down — without flooding. Shipped

A cell is the fundamental unit of organization. Cells form parent/child tiers (node → cell → cohort → federation → coalition) and keep operating autonomously when cut off. The leaf tier is Node in code; see Module 2·5 for the full vocabulary status.

The repositories

Umbrella workspace. peat-lite/

The modularity model (read this twice)

peat-lite and peat-btle are standalone leaf crates. The rest of Peat depends on them; they depend on nothing in Peat. Dependencies flow down into the edge crates, never out of them. Shipped
  • peat-lite is a pure leaf — zero Peat dependencies (only heapless). A self-contained no_std CRDT library.
  • peat-btle is almost a leaf — its only Peat dependency is an optional peat-lite (feature peat-lite-frame). At 0.4.0 it has zero peat-mesh dependency.
  • The core reaches outward to consume them, optionally — peat-mesh pulls in peat-btle (bluetooth) and peat-lite (lite-bridge). Never the other way.
peat-protocol / peat-ffi / peat-mesh   ← the "core" stack
            │  (optional features point DOWN)
            ▼
       peat-btle    ← standalone BLE transport; only optional dep is peat-lite
            │  (optional)
            ▼
       peat-lite    ← pure leaf: zero Peat deps, no_std, reusable anywhere

Why it matters: (1) independent reuse — each edge crate is adoptable on its own; (2) independent versioning/publishing (each is its own crates.io + Maven artifact); (3) it reaches targets the core can't — bare-metal MCUs, mobile; (4) it keeps the graph acyclic. The cycle did appear once: peat-btle used to depend back on peat-mesh; ADR-059 Amendment 4 broke that back-edge (cycle-break tracked by peat-mesh#828, In flight).

Phase 2 — leader election is deterministic

Every node computes the same score for every candidate, so they all converge on the same leader with no vote-counting round. Shipped. Be precise about which formula lives where — there are two:

WhereScoringStatus
peat-protocol cell formation (cell/leader_election.rs)compute·0.30 + communication·0.25 + sensors·0.20 + power·0.15 + reliability·0.10, lexicographic node-id tiebreak. Matches spec-004.Shipped
peat-mesh runtime role (hierarchy/dynamic_strategy.rs)A different formula: mobility, (1−CPU%), (1−mem%), battery%, ×1.1 if can-parent, ×parent-priority; becomes Leader iff its score ≥ best peer × (1+hysteresis).Shipped
Do not present a single global algorithm. The cell-formation weights (0.30/0.25/…) are the spec-004 model; the mesh layer's runtime score uses entirely different inputs. A reader who greps dynamic_strategy.rs will not find the 0.30/0.25 split.

A cell only becomes Ready when CellCoordinator::check_formation_complete() passes: min size, leader elected, roles assigned, capability coverage, readiness ≥ 0.7, and human approval for mission-critical formations — autonomy under human authority.

Phase 3 — the routing rule that defines the topology

peat-mesh/src/routing/router.rs:

// Same cell → always allowed.
if from_cell == to_cell { return true; }
// Cross-cell direct messaging → rejected. Route up through your cell leader.
return false;

That single rule prevents an everyone-talks-to-everyone flood. State flows up (node→cell→cohort→federation→coalition), commands flow down, peers handle handoffs sideways. Cell leaders route upward; non-leaders cannot cross-cell or reach the zone.

State, QoS, security

State is Automerge CRDT documents grouped into named collections; the AutomergeBackend indexes docs by collection:id, fires observer channels on change, and orders deletions with a Lamport-counted tombstone. QoS classifies every message into 5 classes (Critical=1 … Bulk=5) so contact reports are ordered ahead of routine telemetry — ordering ships; cross-class wire-level preemption is In flight. Generic crypto (DeviceKeypair, FormationKey) lives in peat-mesh::security and is re-exported; peat-protocol adds the RBAC policy layer.

Step 2 · Formation auth — the handshake (the security crux)

Membership is proven by a shared formation key, never by mere reachability. FormationKey is derived from formation_id + a 32-byte secret via HKDF-SHA-256. The handshake runs over ALPN b"peat/formation-auth/1" (30 s timeout) before any state is exchanged. Shipped (peat-protocol/src/network/formation_handshake.rs; peat-mesh/src/security/formation_key.rs).

Legend. Solid gray = handshake messages · amber note = constant-time verification · green = admit · red dashed = reject. Fresh nonce + formation_id inside the MAC make it non-replayable; the key never crosses the wire.

Step 3 · Leader election — deterministic, no consensus

Every node computes the same winner from the same advertised capabilities — no vote round, split-brain safe. Shipped. The spec-004 / cell-formation model:

// peat-protocol cell formation (spec-004 model)
score = compute·0.30 + communication·0.25 + sensors·0.20 + power·0.15 + reliability·0.10
ties → lexicographic node id
The mesh runtime uses a different formula. DynamicHierarchyStrategy scores mobility / (1−CPU%) / (1−mem%) / battery% with can-parent and parent-priority multipliers — not the 0.30/0.25 split. Both are deterministic; they just live at different layers.
Election timing constants are illustrative. Numbers like "heartbeat 2 s, re-elect after 3 missed (~6 s), election 5 s" are not traced to a code constant in the audit. peat-node has a reconnect watchdog (5 s interval, 5 s→120 s backoff), but that is a different mechanism. Treat the specific cell-election timings as spec/illustrative until cited.

Step 4 · Roles — Leader elected, the rest assigned

CellRole (capability axis, 7 variants): Leader (elected) + assigned Sensor, Compute, Relay, Strike, Support, Follower Shipped (peat-protocol/src/models/role.rs:14-29). The distinct RBAC Role enum is Leader, Member, Observer, Commander, Admin (the README's "Operator/Supervisor" names are wrong). RoleScorer::best_role_for_platform picks by fitness; required capabilities are blocking.

Result → FormationStatus (Forming / AwaitingApproval / Ready / Failed). Shipped

The CRDT sync wire protocol

Sync is Automerge over Iroh QUIC, reconciled by negentropy set reconciliation. The wire is one-byte-tagged messages in storage/automerge_sync.rs:

#[repr(u8)]

Why negentropy? It is a set-reconciliation protocol that finds the difference between two doc sets in O(log n) rounds by exchanging fingerprints, so peers learn which docs differ before sending heavy data. Shipped (cites arxiv 2012.00472; negentropy = 0.5). The "O(log n)" is the algorithm's claim, not an independent benchmark. Version-vector, snapshot-since-T, hierarchical-digest, and IBLT schemes are not implemented — those are teaching-only design.

The merge rules are small enough to be obvious — CRDTs without the jargon:

// lww.rs — last-writer-wins with a deterministic tiebreaker
fn should_accept(&self, ts: u64, node: NodeId) -> bool {
    ts > self.timestamp
        || (ts == self.timestamp && node.as_u32() > self.node_id.as_u32())
}
// counter.rs — grow-only counter merge = max per node (idempotent, commutative)
pub fn merge(&mut self, other: &Self) {
    for (&node, &c) in other.counts.iter() {
        let e = self.counts.entry(node).or_insert(0);
        *e = (*e).max(c);
    }
}

The 16-byte wire header (ADR-035, Proposed but the codec ships): magic "Peat" (4 ASCII bytes — not 0xCAFE), version 1, message type, flags (2 B LE), node id (4 B LE), seq (4 B LE) — no CRC-16. MAX_PACKET_SIZE = 512 (MAX_PAYLOAD = 496), DEFAULT_PORT = 5555, multicast 239.255.72.76. A Document (0x07) message type carries peat-lite's universal Document envelope (defined in peat-lite/src/protocol/document.rs, not peat_mesh::Document), so a tiny sensor shares the same document model as a server.

The 256 KB budget is a design target, not an enforced limit. ADR-035 states 256 KB, but there is no allocator cap and no static-RAM assertion. The often-quoted "~40× smaller than Automerge" rests on two soft inputs (this target and an unverified ~10 MB Automerge footprint) — treat it as illustrative.

5 · peat-gateway — the control plane

The enterprise side: onboard teams, federate identity, manage keys, and stream mesh events out — without ever sitting in the data path. peat-gateway/

It is a control plane, not a node and not a data plane. peat-gateway implements zero mesh transports, does not sync CRDTs, and does not sit in the peer-to-peer path. External mesh nodes register read-only handles into the gateway's API; the gateway observes and manages. The data-plane "gateway/relay" in DDIL narratives is a peat-mesh relay node, which is a different thing — and its reachability is exactly what DDIL operation must survive.

1 · CoT/TAK operator picture Mostly Shipped

Scenario. Dismounted teams carry phones; a vehicle runs a peat-node sidecar; a CoT consumer at the command post renders the common operating picture.

  • Transports. phones ↔ phones over BLE Shipped; phone ↔ vehicle via a BLE→QUIC bridge (ADR-059 codec Shipped, ADR Proposed); vehicle ↔ command post over QUIC/Iroh Shipped; bridged to the CoT consumer over CoT/TAK TCP Shipped.

2 · UGV / robot joining a cell and being tasked Join Shipped · Tasking In flight

3 · Maritime / beyond-line-of-sight relay Relay Shipped · Satellite Proposed

Scenario. A vessel relays a coastal sensor cell's summary to a shore post; the line-of-sight link drops and it would fall back to satellite.

  • Transports. sensor↔vessel and vessel↔shore over QUIC/Iroh when in range Shipped; the beyond-line-of-sight fallback to Iridium SBD is Proposed (ADR-051) — no crate, no code. A LoRa leg is likewise Proposed (ADR-052).
  • Hierarchy. the sensor cell is a Cell; the vessel is its Leader routing upward; the shore post observes CohortSummary/FederationSummary.
  • Data flow. in range, standard Automerge-over-QUIC with negentropy. Out of LOS, the design intent is to send a compact summary up and commands down over the satellite link using a single-burst digest — but the satellite transport and that digest scheme are both Proposed/Speculative; shipped negentropy assumes an interactive QUIC channel and would not work over a one-shot frame as-is.
  • Worked (clearly Proposed math). a single Iridium SBD frame caps near 1,960 B MO / 1,890 B MT (an Iridium hardware limit from research, not a Peat measurement) at 5–20 s latency, ~$0.04–0.13/msg; a round-trip summary would be ~40 s and two paid messages — illustrative arithmetic over an unbuilt transport. +SBDIX MO+MT-in-one-pass is an external Iridium fact, not a Peat fact.

4 · Disaster-response disconnected cell reconciling on reconnect Shipped

Scenario. A team works in a comms-denied building, accumulates state locally, then reconnects and converges automatically. This is the strongest shipped story in Peat.

  • Transports. intra-team BLE and/or QUIC on local Wi-Fi Shipped; reconnect over QUIC/Iroh. n0 relay is off by default — correct for this air-gapped posture.
  • Data flow. while partitioned, every node commits Automerge changes locally first; a partitioned cell elects a leader locally (deterministic, no quorum → no split-brain stall). On reconnect, the watchdog re-establishes QUIC, negentropy reconciles the document sets, only the genuinely-missing deltas transfer, and Automerge merges deterministically; two independently-elected leaders converge by LWW on leader_id.
  • Caveat. peat-btle reconnect re-delivery of pending CRDT state is In flight (#73); the QUIC/peat-node path is the robust one. Tombstone retention (168 h / 7-day default) governs what survives a long offline window (peat-node#136).
  • Worked. a team offline 30 min accumulates ~200 changes; on reconnect, negentropy exchanges fingerprints in O(log n) rounds (an algorithmic claim, not benchmarked) and transfers only the ~200 missing deltas — not the full history. Convergence is automatic.

5 · Remote sensor field over a constrained link Embedded legs Shipped · LoRa leg Proposed

Scenario. Battery ESP32 sensors report to a better-resourced bridge node, which aggregates into the full Automerge mesh.

  • Transports. sensor↔bridge over peat-lite UDP (codec Shipped; bridged by LiteMeshTransport, feature lite-bridge); bridge↔mesh over QUIC/Iroh Shipped. A LoRa variant of the first leg is Proposed (ADR-052). OTA firmware push is Shipped (lite_ota.rs, ADR-047, stop-and-wait, SHA-256, optional Ed25519 signing).
  • CRDT types on the wire. LwwRegister (latest reading) and GCounter (event counts) — the only core peat-lite CRDTs that actually ship.
  • Hierarchy. each sensor a Node; the bridge node is the Cell leader / boundary translator — a mesh node, not the peat-gateway control plane. A gateway, if present, only observes CDC events.
  • Worked. a reading fits one UDP datagram: 16 B header + ~12 B position (fixed-point lat/lon microdegrees + alt cm) + envelope framing, well under the 496 B payload limit. A GCounter over 32 nodes is ~260 B (a design-budget figure, not measured RAM — the 256 KB total is a design target, ADR-035, not enforced). 9 · The Protocol Specifications

The normative layer — IETF RFC-style specs (docs/spec/, RFC 2119 MUST/SHOULD/MAY). They are the best single description of the protocol's shape, but they have drifted from the code, and the table below lists exactly where.

All specs are Draft (2025-01-07) except spec 005-security, which was amended 2026-05-18 (rev 0.2.0) to AES-256-GCM and is now FIPS-clean. Where a spec disagrees with the code, the code wins.

001 · Transport

Wire formats & lifecycle. QUIC/Iroh primary; stream-id ranges partition control/CRDT/app traffic. UDP bypass is unauthenticated by default — use Signed mode on untrusted links. Drift: spec says DeviceId = "32 bytes of SHA-256" and BLE GATT 0x1826 — both wrong (see table).

002 · Sync

Exclusively Automerge + Negentropy. Automerge uses observed-remove maps and RGA lists/text; map keys resolve last-write-wins by Lamport ts + actor-id tiebreak. Deletes are tombstones (7-day default retention Shipped).

003 · Schema

Protobuf tactical entities in versioned packages; strict evolution rules. Bidirectional CoT/TAK mapping (a-f/a-h/a-n/a-u, <_peat_> extension).

004 · Coordination

Cell state machine, formation handshake, the spec-004 election weights (0.30/0.25/0.20/0.15/0.10), emergent-capability patterns, the three flows, partition merge. Note: the mesh runtime uses a different election formula (Module 2·5).

005 · Security

Ed25519 identity, formation HMAC challenge-response, RBAC + authority levels. FIPS-approved suite: AES-256-GCM, ECDH P-256, HKDF/HMAC-SHA-256. Drift: spec Role enum (Observer/Member/Operator/Leader/Supervisor) ≠ code (Leader/Member/Observer/Commander/Admin); MLS is listed but not implemented.

Eight facts to carry from the specs (corrected)

  • DeviceId = SHA-256(Ed25519 pubkey) truncated to 16 bytes; NodeId is layer-specific (String in mesh, BLAKE3 u32 in btle, bare u32 in lite, iroh EndpointId in node) — not one derivation everywhere.
  • Automerge + Negentropy is the only sync stack; conflict resolution is deterministic.
  • QUIC/Iroh is primary; stream-id ranges partition control/CRDT/app traffic.
  • UDP bypass is unauthenticated by default — use Signed mode on untrusted links.
  • Formation key = HMAC-SHA-256 proof-of-knowledge, separate from the Ed25519 device identity.
  • Leader election is deterministic, no consensus round — but two different scoring formulas exist (cell-formation vs mesh runtime).
  • Emergent capabilities are formal patterns driving capability-based tasking under human authority.
  • Crypto is FIPS-approved (AES-256-GCM / ECDH P-256 / Ed25519 / HKDF / HMAC-SHA-256) but not yet in a CMVP-validated module; the stale ChaCha20/X25519/P-384/MLS references are Proposed-only or doc drift.
